# Verification Boundary and Claim Policy

This repository distinguishes **repository proof**, **profile gate proof**, and **live-controller validation**.
They are not interchangeable.

## 1. Repository proof
Repository proof closes the Python/repository gate only.
Typical evidence includes:

- `python scripts/check_architecture_fitness.py`
- `python scripts/check_protocol_sync.py`
- `python scripts/check_repository_gates.py`
- `VERIFY_PHASE=python ./scripts/verify_mainline.sh`

Repository proof may demonstrate that:

- the repository is structurally consistent,
- canonical contracts are synchronized,
- the selected Python/mainline suites pass in the review environment,
- the mock/control-plane surfaces remain internally coherent.

Repository proof does **not** demonstrate:

- HIL readiness closure,
- production/live xCore SDK binding,
- measured real-controller RT stability,
- clinical or field deployment readiness.

## 2. Profile gate proof
`VERIFY_PHASE=mock|hil|prod ./scripts/verify_mainline.sh` proves only the requested profile gate.
If `ROBOT_CORE_WITH_XCORE_SDK=OFF` or `ROBOT_CORE_WITH_XMATE_MODEL=OFF`, the result is a **contract-shell / build-shell proof**, not a live SDK proof.

That evidence may demonstrate:

- profile-specific configuration closure,
- contract-shell build integrity,
- ctest registration and selected runtime-core tests,
- build-evidence generation in constrained review environments.

It does **not** demonstrate live-controller ownership, network timing, or measured RT behavior unless the run explicitly used the real SDK/model bindings and preserved the evidence bundle.

## 3. Live-controller / HIL validation
Live-controller validation is closed only by the real controller evidence chain, including the HIL checklist and recorded timing/config artifacts.
Use:

- `docs/HIL_VALIDATION_CHECKLIST.md`
- `scripts/run_hil_readiness.sh`
- `scripts/validate_hil_phase_metrics.py`

The HIL item is closed only when the real-controller evidence bundle is archived as a `.zip` with the deployment record, and that archived bundle itself contains `runtime_config.json`, `rt_phase_metrics.json`, and `runtime_readiness_manifest.json`. External readiness files must not be spliced into the proof chain.

## 4. Prohibited wording
The following claims are forbidden unless backed by matching evidence:

- "fully deliverable"
- "production-ready" based only on Python/repository gates
- "HIL validated" without the controller evidence bundle
- "live SDK verified" when the run used `ROBOT_CORE_WITH_XCORE_SDK=OFF`
- "real-time validated" without measured RT metrics from the controller-side evidence chain

## 5. Required reporting format
Every review/delivery summary must separate results into:

- **已静态确认**
- **已沙箱验证**
- **未真实环境验证**

And must state which phase(s) of `verify_mainline.sh` actually ran.


## Environment readiness manifest

The runtime doctor now emits an explicit readiness manifest that separates static/sandbox readiness from real live-runtime verification. `scripts/doctor_runtime.py --manifest-only --write-manifest <path>` must not be interpreted as proof that HIL or robot-side validation has already passed. When a live evidence bundle is used, the readiness manifest must be archived inside that `.zip` bundle; `scripts/write_verification_report.py --live-evidence-bundle ...` must not accept an external readiness file.


## Execution report

For every repository/profile verification run, archive a **verification execution report** generated by `scripts/write_verification_report.py`. The report is the canonical machine-readable summary of executed phases and allowed claims. Human summaries must not exceed the report.
